# 申请表字段规则与验收标准

## 默认填写路径

在用户没有明确覆盖时，按高校普通原创软件登记材料的保守路径填写：应用软件、原创、未发表、独立开发、原始取得、全部权利、由著作权人申请、多种文档交存。

这些默认值不是事实信息。涉及软件名称、日期、权利人、证件号码、联系人和技术环境的字段必须由用户确认。

## 必须向用户确认的字段

| 模块 | 必问字段 | 原因 |
| --- | --- | --- |
| 软件基本信息 | 软件全称、软件简称、版本号、开发完成日期 | 影响表单抬头、页眉、版本号和时间合法性 |
| 技术环境 | 开发/运行硬件环境、开发/运行操作系统、开发工具、支撑软件 | 容易被模型凭经验编造，必须来自项目实际情况 |
| 代码规模 | 源程序量 | 应与项目代码统计一致 |
| 简述字段 | 开发目的、面向领域、主要功能、软件技术特点 | 有字数限制且应贴合项目 |
| 权利主体 | 著作权人名称、类别、证件类型、证件号码 | 涉及权利归属，必须确认 |
| 联系信息 | 申请人名称、联系人、电话、邮箱、手机 | 用于申请办理，必须准确 |

## 字数和数量限制

| 字段 | 限制 | 校验方式 |
| --- | --- | --- |
| 修改软件作品说明 | 100字以内 | 超出时压缩；原创时不填修改说明 |
| 开发目的 | 50字以内 | 超出时改写为一句话 |
| 面向领域/行业 | 50字以内 | 使用行业短语，不写长句 |
| 软件主要功能 | 200字以内 | 覆盖核心功能，不写技术实现细节堆砌 |
| 编程语言 | 最多10项 | 从表单选项中选择；额外语言单独说明 |
| 软件涉及技术特点 | 最多3项 | 当前默认三项已达上限，不再额外增加 |

## 冲突检查

- 选择“未发表”时，不填写首次发表日期和地点；若用户提供发表信息，应改为“已发表”。
- 选择“原创”时，不填写修改说明；选择“修改”时，必须补充修改范围和原软件信息。
- 选择“独立开发”时，不要求合作开发协议；选择“合作开发”时，应提示需要合作开发协议。
- 选择“全部权利”时，不再勾选部分权利；选择“部分”时，必须列明具体权利项。
- 著作权人为单位时，证件类型通常不应是“身份证”；若出现身份证，应标记为需确认。
- 原始粘贴内容中的“证件号码 4”明显不像合法证件号码，必须标记为异常。

## 输出验收标准

合格输出必须满足：

1. 按原表单顺序覆盖所有模块。
2. 每个字段都有填写内容、待补充、不适用或需确认标记。
3. 不用模型臆测权利人、证件号、联系人、日期和代码行数。
4. 明确列出未发表状态下发表信息不适用。
5. 明确列出源程序和文档鉴别材料格式要求：页眉含软件名称和版本号，右上角页码，源程序每页不少于50行，文档每页不少于30行。
6. 对所有字数限制、多选上限和冲突项给出检查结果。
