# 高校软著软件说明书

## 目标

根据软件项目实际功能、运行界面、代码结构和申请表信息，生成可作为软件著作权登记“文档鉴别材料”的软件说明书内容。说明书应解释程序的内容、组成、设计、功能规格、开发情况、测试结果或使用方法，重点产出能直接放进 Word 的完整正文、图注、截图说明和章节展开材料。

默认目标是生成不少于 20000 个中文字符的完整说明书正文，使复制到 Word 后在常规正文排版下可达到 20 页以上。若单次回复无法容纳全文，必须在同一对话中按分卷继续输出，直到达到目标长度并完成验收检查。

## 要求分层

先读取 `references/manual-requirements.md`，区分硬性要求和增强要求：

- 内容依据：软件文档应描述程序内容、组成、设计、功能规格、开发情况、测试结果及使用方法。
- 内容增强：为了降低补正风险、提高文档可信度和完整度，正文要图文并茂，包含多截图、架构图、流程图、功能模块说明、操作步骤和测试结果。
- 排版事项：页眉、页脚、页码和最终 Word 格式只作为提醒，不作为本 Skill 的工作重心。

目录和章节模板见 `references/manual-outline.md`。视觉材料组织规则见 `references/visual-evidence-guidelines.md`。

当用户要求“完整说明书”“一次生成”“20页以上”“长度要保证”“可直接复制到 Word”或未明确要求只要目录/样例时，必须读取 `references/long-form-output-protocol.md` 并按其中的长文档输出协议执行。

## 工作流

### Step 1: 收集输入

先确认以下信息，不足时输出待补充清单：

- 软件全称、版本号、开发完成日期。
- 软件用途、目标用户、核心功能模块。
- 运行入口、主要页面、操作流程。
- 项目技术栈、系统架构、AI/算法模块、数据流。
- 可用截图目录或截图清单。
- 是否已有原型图、流程图、架构图、测试结果或部署说明。

不要虚构真实界面截图。没有截图时，先生成“截图需求清单”和“截图说明文案”，等待用户提供截图或允许制作架构/流程示意图。

如果缺少的信息不会影响说明书主体可信度，例如截图文件暂未提供、测试数据尚不完整，可以使用“图位占位 + 图注说明 + 待替换提示”的方式继续生成正文；不要因为缺截图而只输出目录。

### Step 2: 选择说明书类型

根据软件形态选择文档类型：

| 软件形态 | 推荐文档 |
| --- | --- |
| 有用户界面、平台、网站、小程序 | 用户手册/操作说明书 |
| 后端服务、算法系统、工具库 | 设计说明书 + 使用说明 |
| AI/数据处理系统 | 系统设计说明书 + 功能说明 + 运行示例 |
| 教学/管理/信息安全平台 | 用户手册 + 模块功能说明 + 流程截图 |

如果用户没有指定，默认生成“软件说明书”，内部结构同时覆盖用户手册和设计说明。

### Step 3: 规划内容体量

为了让说明书内容充实，按“不少于 20 页 Word 文档，推荐可扩展到 60 页交存材料”的体量规划章节。不要用空话凑页数，优先通过以下方式扩展内容：

- 每个核心模块写清楚用途、入口、操作步骤、字段说明、输入输出和异常处理。
- 为每个关键流程准备截图位置和图注。
- 为 AI/算法/数据处理部分补充架构图、数据流图和处理流程说明。
- 增加测试场景、错误提示、权限控制、日志/导出等真实功能说明。

如果项目规模较小，仍要优先通过运行环境、模块说明、操作流程、测试场景、异常处理、图表说明和附录扩展到最低目标长度；确实无法达到时，必须说明无法达到的原因和缺少哪些真实材料。

### Step 4: 生成图文并茂内容

按模块生成完整正文。每个核心功能模块至少包含：

1. 模块用途。
2. 用户操作步骤或系统处理流程。
3. 截图/图表占位与说明文字。
4. 输入输出或业务规则。
5. 异常情况或权限限制。

AI 架构图、系统架构图、流程图可以使用 Mermaid 或 ASCII 草图先表达结构，再由用户转成图片；图必须基于真实架构，不要画与项目无关的装饰图。

完整说明书输出时，先给出“文档生成进度”和“本次输出范围”，再连续输出正文。分卷输出时使用“第 1 卷/第 2 卷/第 3 卷”等标题，并在每卷末尾标明累计章节、估算字符数和下一卷起点。

### Step 5: 内容验收检查

输出或生成文档后检查：

- 软件名称、版本号、功能定位与申请表一致。
- 文档类型可被解释为程序文档，例如用户手册、操作手册、设计说明书、流程图说明。
- 章节覆盖软件概述、运行环境、系统设计、功能模块、操作流程、测试与异常处理。
- 截图、架构图、流程图都有编号、标题和说明。
- 每个核心功能模块都有具体功能说明和操作/处理流程。
- AI 架构图或系统架构图与项目真实模块一致。
- 文档内容覆盖软件功能、使用方法、设计/组成、开发环境和测试结果中的多个维度。
- 完整说明书请求下，正文估算中文字符数不少于 20000；若不足，继续补写未充分展开的章节。
- 不包含无关模板广告、代理机构话术、售卖模板痕迹。

## 输出格式

根据用户需求选择输出：

- `说明书撰写要求总结`：先明确内容要求和实操增强要求。
- `说明书目录`：生成章节结构和页数规划。
- `截图清单`：列出每页或每模块需要的截图/图表。
- `正文草稿`：按章节生成可粘贴进 Word 的内容。
- `验收清单`：检查章节覆盖、功能完整度、截图/图表说明和一致性。

默认输出完整说明书正文，而不是只输出样例。只有当用户明确要求“先给目录”“先给计划”“只要样例”“先不要写全文”时，才输出“目录 + 截图清单 + 前两章正文样例 + 验收清单”。

完整说明书使用以下结构：

```markdown
## 说明书生成进度

- 目标长度：不少于 20000 中文字符，复制到 Word 后按 20 页以上规划
- 当前卷次：第 N 卷
- 本卷范围：...

## 软件说明书正文

...

## 本卷自检

- 已覆盖章节：...
- 估算累计字符数：...
- 下一卷起点：...
```

## 禁止事项

- 不要引用或模仿模板售卖站的营销话术。
- 不要承诺“包过”“一定通过”。
- 不要使用与项目无关的通用截图或虚假界面图。
- 不要用空泛套话凑页数；每段内容都要能对应实际功能、界面、代码模块或处理流程。
- 不要把 Word 页眉页脚、页码等最终排版事项当成本 Skill 的主要工作。
- 不要在完整说明书请求中只输出大纲、目录或少量样例，除非用户明确要求先暂停在规划阶段。
