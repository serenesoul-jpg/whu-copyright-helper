# 正文去 AI 化写作规则

用于生成软件说明书正文、用户手册、设计说明书、操作手册和长文档分卷时。目标不是写成文学化文本，而是写成像项目成员整理出的真实软件材料。

## 总体原则

- 使用朴素、明确、可核验的项目文档语气。
- 以软件实际功能、页面、接口、数据、角色、权限、日志、测试和异常处理展开。
- 每段尽量回答一个具体问题：该模块做什么、谁使用、入口在哪里、输入什么、系统如何处理、输出什么、异常如何提示。
- 允许适度使用通用背景，但不要把通用背景当成主要篇幅。
- 不要写“本文将深入探讨”“在数字化浪潮下”“具有重要意义”“显著提升”“保驾护航”“赋能”等泛化表达。

## 推荐写法

| 场景 | 推荐写法 |
| --- | --- |
| 模块说明 | “用户在登录页面输入账号和密码，系统先校验字段是否为空，再向后端提交认证请求。” |
| 字段说明 | “文档标题为必填项，长度建议控制在 100 字以内；为空时系统在输入框下方显示提示。” |
| 流程说明 | “管理员进入分类管理页面后，可以新增、编辑或停用分类。停用分类不会删除历史文档。” |
| 异常说明 | “当上传文件超过大小限制时，系统阻止提交并提示用户压缩文件或拆分上传。” |
| 图注说明 | “图 5-2 展示文档上传页面，左侧为文件选择区域，右侧为元数据录入区域。” |
| 测试说明 | “测试人员使用普通成员账号访问权限配置页面，系统应拒绝访问并返回权限不足提示。” |

## 避免写法

避免以下高频 AI 腔：

- “随着信息技术的快速发展……”
- “本系统具有良好的可扩展性、稳定性和安全性。”除非后文说明具体机制。
- “大大提高了工作效率”“极大地方便了用户”“为用户提供了良好体验”。
- “通过先进的人工智能技术实现智能化管理”。如果确有 AI 功能，写清楚输入、模型或规则、输出和人工复核方式。
- 连续多段使用“首先、其次、最后”“一方面、另一方面”“不仅……而且……”这类模板句式。
- 使用与项目无关的功能补篇幅，例如没有报表导出却写大量导出说明。

## 段落密度

- 每个二级小节至少包含 2-4 个自然段，避免只有一句口号。
- 每段 120-250 字较合适；超过 350 字时拆分。
- 说明书正文可以使用表格，但表格后必须配解释文字，不能只堆表格。
- 图位占位后必须写 150-300 字说明，解释图中区域、操作目的和系统反馈。

## 细节优先级

扩展正文时按以下顺序补充细节：

1. 页面或入口：用户从哪里进入该功能。
2. 操作角色：管理员、普通用户、审核人员或系统服务。
3. 输入数据：字段、文件、参数、筛选条件或配置项。
4. 处理规则：校验、权限判断、状态流转、数据写入、模型推理或任务调度。
5. 输出结果：列表、详情、报告、提示、日志、导出文件或状态变化。
6. 异常情况：空值、重复、权限不足、网络失败、文件过大、格式错误、任务失败。
7. 验证方式：测试用例、预期结果、人工复核点。

## 真实性边界

- 不要虚构真实截图、真实测试结果、真实用户数量、真实性能指标或真实部署地址。
- 没有截图时，使用“图位占位 + 图题 + 图注说明”，并标注后续替换真实截图。
- 不确定的项写成“待确认”或“可根据实际部署环境填写”，不要包装成事实。
- 默认不要写具体学校、学院、证件号码、联系人、手机号、邮箱和地址。

## 自检清单

完成说明书正文或每个分卷后检查：

- 是否有连续两段以上泛泛描述而没有功能细节。
- 是否出现模板化开头、宏大背景、营销话术或保证性表达。
- 每个核心模块是否包含入口、角色、步骤、输入、输出和异常处理。
- 图位和表位是否都有解释性文字。
- 待确认信息是否明确标出，没有被写成事实。
- 文风是否像项目说明文档，而不是宣传稿或 AI 总结稿。
