# 长文档输出协议

用于用户要求完整软件说明书、20 页以上 Word 文案、一次对话生成全文、长度要保证或可直接复制到 Word 时。

## 长度目标

默认生成不少于 20000 个中文字符的正文内容。该目标按常见 Word 正文排版估算，通常可支撑 20 页以上内容；实际页数仍受字号、行距、页边距、截图尺寸和目录占位影响。

如果用户明确要求更高页数，按以下估算提高目标：

| 目标页数 | 建议正文长度 |
| --- | --- |
| 20 页 | 20000-25000 中文字符 |
| 30 页 | 30000-36000 中文字符 |
| 60 页 | 55000-70000 中文字符，通常需要分多轮生成 |

正文长度不包含封面、目录、纯 Markdown 表格分隔符、简短进度提示和验收清单。图题、图注、截图说明、流程说明和测试说明计入正文。

## 章节字数预算

生成 20 页以上说明书时，按以下最低预算规划：

| 章节 | 最低中文字符数 | 展开重点 |
| --- | ---: | --- |
| 1. 引言 | 1800 | 编写目的、适用对象、术语定义、参考资料、文档约定 |
| 2. 软件概述 | 2600 | 背景、建设目标、用户角色、应用场景、业务价值 |
| 3. 运行环境 | 2200 | 硬件、操作系统、浏览器/依赖、部署、网络和数据要求 |
| 4. 系统总体设计 | 3200 | 架构、模块、数据流、接口、权限、安全、AI/算法链路 |
| 5. 功能模块说明 | 8000 | 每个核心模块的用途、入口、操作步骤、字段、输入输出、异常处理 |
| 6. 操作流程 | 3000 | 从登录到核心任务完成、查询、导出、配置和退出的完整流程 |
| 7. 测试与异常处理 | 2400 | 测试环境、测试用例、边界场景、错误提示、权限校验 |
| 8. 维护与附录 | 1800 | 日常维护、日志、备份、常见问题、版本说明、图表索引 |

如果项目模块较多，把第 5 章扩展为主要篇幅来源。每个核心功能模块不少于 1200 中文字符；少于 4 个模块时，应把单模块展开到 1800-2500 中文字符。

## 完整输出流程

1. 先用 5-8 行列出“说明书生成进度”，说明目标长度、计划卷数、本卷范围和是否需要占位图。
2. 直接输出可复制进 Word 的正文。不要把主要篇幅花在解释写作方法上。
3. 按章节连续展开，不要只给目录或提纲。章节标题使用 Word 友好的编号，例如 `1. 引言`、`1.1 编写目的`。
4. 每个图位使用可替换占位，例如 `图 4-1 系统总体架构图`，随后写 150-300 字说明。
5. 每个表位使用 `表 7-1 功能测试用例表` 这类标题，并配说明文字；表格可以简洁，但说明文字要完整。
6. 每卷末尾输出“本卷自检”，包含已覆盖章节、估算累计字符数、仍需补写章节和下一卷起点。
7. 如果估算累计字符数不足 20000，继续输出下一卷，不要宣布完成。

## 分卷规则

一次回复放不下全文时，使用分卷输出。推荐分卷：

| 卷次 | 覆盖内容 | 目标字符数 |
| --- | --- | ---: |
| 第 1 卷 | 封面信息、目录、1-3 章 | 6000-8000 |
| 第 2 卷 | 4 章总体设计、5 章前半功能模块 | 7000-9000 |
| 第 3 卷 | 5 章后半、6-8 章、验收清单 | 8000-10000 |

如果模型或平台停止在中途，恢复时从上一卷“下一卷起点”继续，不要重写已经完成的章节。

## 内容填充规则

扩展篇幅时优先增加真实、有用的说明：

- 模块入口、角色权限、前置条件、操作步骤、页面字段、数据来源、处理结果、异常提示。
- 架构图说明、数据流说明、模型/规则处理链路、接口调用关系、日志和审计机制。
- 测试用例、边界条件、错误输入、权限不足、网络异常、数据为空、导出失败等场景。
- 截图图注、图中区域解释、用户在该页面完成的任务、系统校验和反馈。

不要通过重复同义段落、泛泛价值表述、模板营销话术或与项目无关的功能凑字数。

## 长度自检

完成时必须输出：

```markdown
## 完整性与长度自检

- 估算正文中文字符数：约 XXXXX 字
- 预计 Word 页数：约 XX-XX 页，按正文 + 图表占位估算
- 已覆盖章节：1-8 章完整 / 部分章节待补
- 图表占位数量：图 X 个，表 X 个
- 仍需用户替换或确认的材料：...
```

估算字符数明显不足时，继续补写薄弱章节，优先补第 5 章功能模块说明和第 6 章操作流程。
