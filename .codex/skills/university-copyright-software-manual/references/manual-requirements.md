# 软件说明书要求核验

## 来源依据

本 Skill 的硬性要求来自以下可靠来源：

- 中国版权保护中心《软件登记指南》（官网入口：`https://www.ccopyright.com.cn/index.php?optionid=1030`）。
- 用户提供 zip 中的《附件1：软件著作权登记申请所需文件.doc》。
- 用户提供 zip 中的《附件2：计算机软件著作权登记申请表填写说明.doc》。
- 用户提供 zip 中的《附件3：软件著作权登记常见问题问答.doc》。

不要把模板售卖站、代理机构营销页、泛泛博客当作硬性依据。

## 内容要求

| 项目 | 要求 |
| --- | --- |
| 软件文档定义 | 文档是描述程序内容、组成、设计、功能规格、开发情况、测试结果及使用方法的文字资料和图表，如程序设计说明书、流程图、用户手册等。 |
| 文档种类 | 可提交一种文档，也可提交一种以上文档。申请表中选择多种文档时，应列明种类。 |
| 内容范围 | 至少覆盖软件功能、设计组成、运行环境、使用方法、测试结果或开发情况中的多个维度。 |
| 文字与图表 | 可以使用文字资料和图表共同说明软件，图表包括流程图、架构图、用户手册截图等。 |

## 排版与交存提醒

这些是最终 Word/PDF 材料处理时要注意的格式事项，但不是本 Skill 的主要工作：

- 一般交存通常提交文档前、后各连续30页；不足60页的全部提交。
- 文档通常需要在页眉标注软件名称和版本号，并在右上角标注页码。
- 文档页通常要求每页不少于30行。
- 常见问答建议文档从目录开始由第1页编到第60页。

## 实操增强要求

这些不是官网逐字硬性条款，但适合作为高校软著材料的增强质量标准：

- 图文并茂：每个核心功能模块配截图、流程图或架构图，并配有解释性文字。
- 截图充分：优先覆盖登录/首页、核心业务流程、数据输入、结果展示、管理/配置、异常提示、导出/报告等页面。
- 图有说明：每张图必须有编号、标题和文字解释，避免只有图片没有正文。
- AI 架构图：如果项目涉及 AI、算法、模型调用或智能分析，加入真实的数据流、模型处理流程、推理/分析模块图。
- 页数稳妥：内容体量按可扩展成约60页规划，但不为了页数制造无意义重复。
- 内容真实：所有截图和架构说明必须能对应实际系统、原型或代码功能。

## 常见误区

- “必须全是截图”是错的。说明书是文字资料和图表，截图应服务于说明。
- “随便买模板改名”风险很高。模板痕迹、功能不对应、截图不真实都可能导致补正。
- “页数越多越好”不准确。关键是内容与软件一致、功能覆盖完整、截图和图表能说明真实系统。
- “AI 架构图可以随便画”不准确。架构图应反映真实模块、数据流和技术栈。
