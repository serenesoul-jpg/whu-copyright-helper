# Skill 常见反模式

## 一个 Skill 做太多事

表现：同一个 Skill 同时负责代码审查、迁移、部署和文档生成。

修正：按任务拆分。只有当多个步骤确实属于同一条工作流时，才用一个精简的主 Skill 做编排。

## Description 太模糊

表现：

```yaml
description: 处理迁移。
```

修正：写清楚来源、目标、语言、文件类型、关键操作和排除场景。

```yaml
description: >
  将 Go 代码从 old-http-client 迁移到 unified-httpclient。
  用于 import 替换、请求构造更新、响应处理变更和迁移验证。
  不要用于一般代码审查或 HTTP 运行时问题调试。
```

## 只有内部黑话

表现：description 依赖团队缩写，外部 Agent 无法判断含义。

修正：内部术语后补充通用解释和技术关键词。

## 没有示例

表现：Skill 写了很多规则，但没有 Before/After。

修正：补 3-5 个紧凑示例，覆盖典型、变体和边界场景。

## 没有检查点

表现：流程连续跑完五步，最后才验证。

修正：每个关键阶段后增加一个小的验证命令或检查清单。

## 写死魔法数字

表现：强制规定某个超时、批量大小或重试次数，却没有上下文。

修正：给出判断规则和安全范围，而不是只给固定值。

## 把 SKILL.md 写成 Wiki

表现：历史背景、架构叙述、政策长文淹没了真正的操作步骤。

修正：背景资料放进 `references/`，`SKILL.md` 只保留做什么、怎么做、怎么验证。

## 不安全自动化

表现：脚本直接删除文件、改数据库或调用接口，没有确认、备份或回滚。

修正：校验输入，预览变更，对危险操作要求确认，并写清楚回滚方式。
