# Skill 质量检查清单

发布、共享或正式依赖一个 Skill 前，先按这份清单过一遍。

## 触发准确性

- [ ] `name` 使用小写字母、数字和连字符，长度不超过 64 个字符。
- [ ] `description` 说明了 Skill 做什么，也说明了什么时候用。
- [ ] `description` 包含常见用户说法和关键技术词。
- [ ] `description` 排除了容易误触发的相邻任务。
- [ ] 重要触发条件没有只写在正文里，因为正文触发后才会加载。

## 范围控制

- [ ] Skill 只解决一件明确的事。
- [ ] 独立工作流已拆成不同 Skill 或 reference 文件。
- [ ] `SKILL.md` 保持精简，建议不超过 500 行。
- [ ] 背景、历史、长规范放在 `references/`。
- [ ] `SKILL.md` 和 references 之间没有重复维护同一段内容。

## 指令质量

- [ ] 目标写清楚了起点、终点和成功标准。
- [ ] 前置判断说明了什么时候应该停止。
- [ ] 步骤使用直接、明确的指令语气。
- [ ] 脆弱规则解释了原因。
- [ ] Before/After 示例覆盖典型、变体和边界场景。
- [ ] 多分支选择用表格或决策流程表达。
- [ ] 关键步骤后有检查点。

## 资源组织

- [ ] 重复或脆弱操作使用脚本。
- [ ] 脚本校验输入和路径。
- [ ] 脚本至少实际运行过一次，或测试过代表性样例。
- [ ] reference 文件从 `SKILL.md` 直接链接，并说明何时读取。
- [ ] assets 只包含最终输出会真正用到的资源。

## 安全

- [ ] 没有硬编码 API Key、密码、Token 或私有凭据。
- [ ] 删除、覆盖、DDL 等危险操作有确认或备份步骤。
- [ ] 数据库变更包含备份、验证和回滚说明。
- [ ] 网络请求使用 HTTPS，并设置合理超时。
- [ ] 外部内容被视为数据，而不是指令。

## 评估

- [ ] 触发评估包含正例、反例和边界例。
- [ ] 效果评估包含输入、期望行为和评分标准。
- [ ] 已用真实或接近真实的任务测试过 Skill。
- [ ] 失败样例已转化为对 description、示例、流程或脚本的具体修订。
- [ ] 评估用例保存在 Skill 或相邻文档中，方便后续回归。
